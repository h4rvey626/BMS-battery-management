# BMS_s940 接线与 GPIO 占用说明

## 1. 说明

本文档根据当前工程代码整理，主要依据以下文件：

- `Project/main.c`
- `BSP/IO_CTRL.c`
- `BSP/led.c`
- `BSP/can.c`
- `BSP/usart.c`
- `BSP/usart2.c`
- `BSP/i2c1.c`
- `BSP/BQ76930.c`

说明原则：

- 以当前 `main.c` 实际初始化和调用到的外设为准。
- 代码中存在但未在 `main.c` 启用的接口，单独列为“预留/未使用”。
- 个别控制脚仅能根据命名推断用途，已在备注中标明。

## 2. 当前已启用的接口

`main.c` 中当前实际初始化的接口如下：

- `USART1`
- `USART2`
- `LED`
- `IO_CTRL`
- `I2C1`
- `CAN1`
- `BQ76930`

未在当前主程序中初始化的模块：

- `I2C` (`PB6/PB7`)
- `I2C2` (`PB10/PB11`)
- `SPI/W25QXX`

结论：

- `PA11/PA12` 当前用于 `CAN`
- `PA3` 当前用于 `USART2_RX`
- `W25QXX` 驱动文件虽然存在，但当前固件没有调用 `W25QXX_Init()`，因此当前版本中可以认为 `W25QXX` 没有被实际使用

## 4. 分模块接线说明

### 4.1 BQ76930

当前代码中，`BQ76930` 通过软件 `I2C1` 与 MCU 通信，并使用单独唤醒脚。

接线如下：

- `PB8 -> BQ76930 SCL`
- `PB9 -> BQ76930 SDA`
- `PA8 -> BQ76930 WAKE`

补充说明：

- `PB8/PB9` 被配置为开漏输出，硬件上通常需要外部上拉电阻。
- `BQ76930.c` 中明确通过 `MCU_WAKE_BQ` 拉高再拉低的方式唤醒器件。

### 4.2 CAN 总线

MCU 侧接线如下：

- `PA12 -> CAN_TX`
- `PA11 -> CAN_RX`

通常完整连接关系应为：

- `PA12 -> CAN 收发器 TXD`
- `PA11 -> CAN 收发器 RXD`
- 收发器再连接到总线 `CANH/CANL`

注意：

- `PA12` 在 `IO_CTRL.h` 里还定义过 `MCU_KZ_P2`，但当前程序随后会将其重新配置为 `CAN_TX`，因此运行时应以 `CAN` 功能为准。

### 4.3 USART1

代码中 `USART1` 波特率初始化为 `115200`，并通过 `UartSend()` 持续发送显示指令和数据。

接线如下：

- `PA9 -> 外部设备 RX`
- `PA10 -> 外部设备 TX`

从发送内容判断，`USART1` 大概率接的是串口屏或上位机显示端。

### 4.4 USART2

代码中 `USART2` 配置为 `9600`，注释写明用于蓝牙串口。

接线如下：

- `PA2 -> 蓝牙模块 RX`
- `PA3 -> 蓝牙模块 TX`

这也说明 `PA3` 当前已经被 `USART2_RX` 占用，不能再给 `SPI/W25QXX` 使用。

### 4.5 LED

板载 LED 接线如下：

- `PA15 -> LED1`
- `PB13 -> LED2`
- `PB14 -> LED3`
- `PB15 -> LED4`

补充说明：

- 工程中关闭了 `JTAG`、保留 `SWD`，因此 `PA15`、`PB3`、`PB4` 可作为普通 GPIO 使用。

### 4.6 IO 控制脚

以下引脚在 `IO_CTRL_Config()` 中被初始化，但仅从代码命名无法 100% 确认外部器件，需要结合原理图最终确认：

- `PA0 -> MCU_KZ_QB_POWER`
- `PA1 -> MCU_D_POWER`
- `PA12 -> MCU_KZ_P2`，但当前程序运行时会被 `CAN_TX` 接管
- `PB12 -> MCU_IN_STA1`
- `PB1 -> STA_WAKE`
- `PB2 -> MCU_BQ1_ART`
- `PB3 -> 预留控制输出`
- `PB4 -> 预留控制输出`
- `PB5 -> 预留控制输出`

其中默认电平：

- `PB12` 初始化后被置高
- `PA12` 若按 `IO_CTRL` 初始化会被拉低，但随后会被 `CAN_TX` 接管
- `PB3/PB4/PB5` 初始化后被拉低

### 4.7 与 `IO_CTRL.h` 的核对结果

`IO_CTRL.h` 中定义的 GPIO 如下：

- `PA0 -> MCU_KZ_QB_POWER`
- `PA1 -> MCU_D_POWER`
- `PA8 -> MCU_WAKE_BQ`
- `PA12 -> MCU_KZ_P2`
- `PB12 -> MCU_IN_STA1`
- `PB1 -> STA_WAKE`
- `PB2 -> MCU_BQ1_ART`
- `PB3 -> PB3`
- `PB4 -> PB4`
- `PB5 -> PB5`

与本文档核对结果：

- 上述 10 个引脚已全部在本文档中体现，没有遗漏。
- 唯一需要特别说明的是 `PA12`：它在 `IO_CTRL.h` 中命名为 `MCU_KZ_P2`，但当前固件启动后又被 `CAN` 驱动重新配置为 `CAN_TX`，因此实际运行状态应以 `CAN_TX` 为准。
- `PA11`、`PA9/PA10`、`PA2/PA3`、`PA15`、`PB8/PB9`、`PB13/PB14/PB15` 不在 `IO_CTRL.h` 中，是由 `CAN`、`USART`、`LED`、`I2C` 等其他驱动单独管理，这与本文档描述一致。

## 5. 预留但当前未使用的接口

### 5.1 预留 I2C

代码中还保留了两组软件 `I2C` 定义，但当前 `main.c` 没有调用：

- `PB6 -> I2C SCL`
- `PB7 -> I2C SDA`
- `PB10 -> I2C2 SCL`
- `PB11 -> I2C2 SDA`

因此当前固件中，这四个脚不属于已启用接口。

### 5.2 SPI / W25QXX

工程中存在 `spi.c` 和 `w25qxx.c`，但当前 `main.c` 未调用以下函数：

- `SPI2_Init()`
- `W25QXX_Init()`

因此当前固件中，`W25QXX` 没有被实际使用。

另外，代码里这部分还存在引脚定义冲突，说明它更像是移植残留或预留代码：

- `PA3` 已被 `USART2_RX` 占用
- `PA4` 同时被写成 `W25QXX_CS`
- `spi.c` 中的 SPI 引脚定义与常规 `SPI2` 管脚也不一致

所以在没有重新梳理代码前，不建议按现有 `SPI/W25QXX` 文件直接接线。

## 6. 当前建议采用的实际接线结论

如果只按当前这版程序运行，建议按下面理解接线：

- `BQ76930`: `PB8/PB9/PA8`
- `CAN`: `PA11/PA12`
- `串口屏或上位机`: `PA9/PA10`
- `蓝牙串口`: `PA2/PA3`
- `LED`: `PA15/PB13/PB14/PB15`
- `其他控制脚`: `PA0/PA1/PB1/PB2/PB12/PB3/PB4/PB5`

## 7. FreeRTOS 7 任务扩展建议

如果后续硬件扩展为：

- `BQ76940` 负责电芯监测、保护、均衡、充放电控制接口
- `BQ34Z100-G1` 负责 `SOC/SOH/循环次数/容量学习`
- `BQ76200`（如果你说的 `B076200` 指的是它）负责主功率 `MOS` 驱动
- `CAN` 负责上位机通信或多 MCU 通信

那么将系统拆分为 `7` 个 `FreeRTOS` 任务是合理的，比当前仅 `BQ76930 + CAN` 的方案更适合做细分任务。

建议的 7 任务划分如下：

| 任务 | 主要职责 | 主要器件/接口 | 备注 |
| --- | --- | --- | --- |
| `sample_task` | 采集单体电压、总压、电流、温度、故障状态、充放电状态 | `BQ76940` | 周期性高频采样，是保护和上报的数据源 |
| `protect_task` | 处理 `OV/UV/OCD/SCD/OT/UT`，执行故障响应 | `BQ76940` + `BQ76200` | `BQ76940` 负责检测和保护寄存器，`BQ76200` 负责主功率回路执行 |
| `balance_task` | 按压差和策略启停均衡 | `BQ76940` | 均衡开关通常由 `BQ76940` 控制 |
| `fuel_task` | 计算 `SOC/SOH`、循环次数、剩余容量、容量学习 | `BQ34Z100-G1` | 这是油量计芯片最适合独立承担的任务 |
| `logic_task` | 工作模式切换、休眠唤醒、充放电策略、故障恢复、系统状态机 | `MCU` | 负责综合调度各芯片数据和动作 |
| `can_task` | `CAN` 收发、协议封装、上位机/多 MCU 通信 | `CAN` | 可同时承担节点间状态同步 |
| `service_task` | 参数存储、日志、升级、串口调试、看门狗、自检维护 | `Flash/UART` | 低优先级后台任务 |

### 7.1 `BQ76940` 负责的任务

如果按上面的 7 任务来拆，`BQ76940` 主要负责以下几类任务：

- 直接负责 `sample_task`
  - 采集单体电压
  - 采集总压/电流相关寄存器
  - 采集温度通道
  - 读取系统状态和故障状态
- 直接负责 `protect_task` 的检测侧
  - 提供 `OV/UV/OCD/SCD/OT/UT` 等保护判断依据
  - 提供 `CHG/DSG` 状态寄存器
  - 参与充放电开关控制
- 直接负责 `balance_task`
  - 控制均衡相关寄存器
  - 执行单体均衡开关动作
- 间接参与 `logic_task`
  - 作为系统状态机的核心输入源
  - 为休眠/唤醒、故障恢复、模式切换提供实时状态
- 间接参与 `can_task`
  - 为 `CAN` 上报提供底层测量数据和故障状态

### 7.2 其他两个芯片的建议职责

- `BQ34Z100-G1`
  - 更适合承担 `SOC/SOH/循环寿命/容量学习` 相关任务
  - 不建议用它替代 `BQ76940` 去做电芯级保护
- `BQ76200/B076200`
  - 更适合承担主功率 `MOS` 的驱动执行
  - 它更像“执行器/驱动器”，而不是独立的数据采集核心

### 7.3 是否值得拆成 7 任务

在以下条件下，拆成 `7` 个任务是合适的：

- 不再只是单一 `BQ + CAN` 的简单轮询系统
- 加入 `BQ34Z100-G1` 做独立油量计
- 加入 `BQ76200` 做主回路驱动
- `CAN` 既要对上位机通信，又要承担多节点或多 MCU 交互
- 后续还要扩展升级、日志、维护、自检等后台能力

如果只是维持当前这版工程功能，那么没有必要强行拆到 `7` 个任务。

## 8. 后续建议

如果后面你已经有原理图，建议再补充一版“引脚 -> 连接器/芯片引脚号 -> 网络名”的正式硬件接线表，例如：

- `PA8 -> BQ76930 WAKE`
- `PA11 -> CAN收发器 RXD`
- `PA12 -> CAN收发器 TXD`
- `PA2/PA3 -> 蓝牙模块`

这样以后维护代码和排查硬件会更方便。
